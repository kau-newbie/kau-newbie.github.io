# Hadoop 공부 정리

**서론**

hadoop은

- HDFS
- MapReduce

로 이루어져있다. 모두 대규모 분산 시스템을 이룬다. 즉, 정말 많은 수의 노드(이하 '서버'와 혼용.)가 서로 Network(이하 NW)로 이루어져 하나의 거대한 시스템을 이루고 있다는 뜻이다.

여기서 둘은 모두, 중앙에서 통제하는 master node들과, 통제에 따라 일을 수행하는 slave nodes로 이루어져있다.
- 요즘은 용어가 좀 그래서 leader - follower node라고 부르기도 한댄다.
- 엄밀히는 각각 Namenode - Datanode, Jobtracker - Tasktracker 라고 부른다.
    - job은 유저가 요청한 일의 단위이다. 여러 task(더 작게 나눈 일의 단위)로 나눌 수 있다. 이 말인 즉슨, 일련의 여러 과정들을 포함할 정도로 큰 작업이다.
- 보통 `namenode`와 `jobtracker` 이 2개를 같이 하나의 노드에 넣어둔다. 하나의 노드에서 돌아가게 한다.

하둡이 좋은 이유는, 아래에서 반복하겠지만, 우리가 신경써서 정해야할(설계 및 프로그래밍) 부분을 매우매우 줄여준다는 것이다.

## 일반적인 빅데이터 처리 과정

배운 바로는, 빅데이터 처리 과정은 다음과 같다.

1. 정말 많은 수의 records(저장된 데이터)들에 대해 반복 작업을 수행한다.
2. **records들에게서 우리가 관심 있는 부분만 추출한다.** `map`
3. 분산 노드들 사이에서 중간 결과들이 정렬되고, 이동한다.
4. **중간 결과를 합쳐준다.** `reduce`
5. 최종 결과물을 출력한다.

여기서 우리는 2번과 4번, 즉 각각 "map"과 "reduce"만 구현해준다. 

그 외 나머지는 하둡이라는 runtime framework가 해줄 것이다.

## Map-Reduce 과정

하둡에서 개발자인 우리는 기본적으로 map()과 reduce()만 정의하면 된다고 한다.

### Map 과정

1. 사용자의 인풋은 master node로 가고, 다시 master node의 지시가 mapper에게 전달된다. 
- mapper는 map()을 수행하도록 master node에게 명령 받은 노드이다.
- map 프로그램 자체가 데이터가 있는 노드로 간다.
- 여기서 저장된 데이터 파일은 'split'이라는 단위로 읽힌다.
2. mapper에서는
    1. **map**을 수행한다. 의미 있는 key값과 그에 따른 value로 (<k,v>) domain(데이터 형태)이/가 변경된다.
    2. map()과정의 중간 결과물들은 mapper 노드의 local buffer에 차곡차곡 저장된다.
    3. buffer의 80% 정도가 차면, spill.out 이라는 형태의 파일로 디스크에 저장한다.
        - 디스크로 보내기 전, 버퍼에서는 `partition`과 `local sort`를 수행한다.
        - partition: 중간 결과를 몇 개로 쪼갤 것이냐? 
            - 보통 reducer의 갯수만큼 쪼갠다. 설정을 따로 할 수 있다.
            - partition들은 하나의 spill파일 안에 들어간다. 나중에 reducer의 요청에 해당하는 spill 파일 일부(특정 partition!)를 읽어 보내준다.
        - local sort: 같은 파티션 안에는 여러 개의 key-value 쌍들이 있다. 이 <k,[v]>들을 정렬해준다.
    4. combiner를 미리 설정해놓았으면, spill 파일들을 정렬한 후, 합쳐 하나의 map output file로 출력한다.
    - '할 수 있으면'(예를 들어, reduce()과정을 통해 특정 key값"apple"을 가진 데이터 value들의 합(1+1+...)을 보고 싶은 경우) 미리 어느정도 k에 대한 v들을 요약한다. (<"appe",15>처럼 합을 구하는 등...)
    5. reducer의 요청에 따라 **shuffle**(network를 타고 partition을 지정된 reducer node로 copy하기.)
    - 이때! shuffle은 각 map output file이 완성될 때마다 개별적으로 수행하게 된다.
    - 모든 input을 처리할 때까지 (shuffle을 하지 않고) 기다리는 건, reducer를 기다리게 만드는 일이므로 낭비이기 때문이다.

### Reduce 과정

1. copy된 map의 output(partition 단위)들이 속속들이 NW를 타고 정해진 Reducer에게 도착한다.

2. Merge&Sort: reducer에 메모리 버퍼에 받아놓은 여러 partition들을(+spill 파일들도) 정렬하고 하나로 크게 합친다.
- 똑같이 버퍼가 넘치면 디스크에도 쓴다. (spill 파일이 mapper와 마찬가지로 존재한다.)
- 각 reducer는 combiner를 통해 (combiner는 reducer에게도 있다!) '할 수 있으면'(예를 들어, reduce()과정을 통해 특정 key값을 가진 데이터 value들의 합을 보고 싶은 경우) 미리 어느정도 같은 key에 대해 <k,v>들을 병합 한다.

2. Grouping: 정렬된 데이터에서 같은 키($k$)를 가진 값($v$)들을 하나의 리스트(Iterable)로 묶음.

3. Reduce: 미리 사용자가 짜놓았던 reduce(k, values) 함수에 데이터를 넣어줌.

4. 출력 (<k,v>)

### 간단한 예시 - word count

- String docid: document.txt 파일의 id
- String text: 실제 txt 파일의 내용

```java
Map(String docid, String text):
    for each word w in text:
        Emit(w,1); // reducer에게 보낸다.
        /**
         * 예시:
         * Map()을 거치면,
         *  ("apple",1), ("apple",1), ..., ("love",1)... 와 같은 output 생성
         * 
         * 이제 shuffle&sort 단계에서
         *  - 위에서 배운 buffer(partitioning & local sort) -> local disk -> combined map output file
         *  - local sorting을 거친 뒤, 새로운 domain을 가지게된 <k,v> 데이터는 ("apple",[1,1,1,1],... ) 형태일 것이다.
         * 마지막으로 shuffle로 각 partition 단위로 읽어 reducer에게 보낸다.  
         * 
         * Rduce()의 인자로,
         * - term = (sorted [<k,[v]>]).k
         * - values = (sorted [<k,[v]>]).v
         * 이렇게 넘겨준다.
         **/

Reduce(String term, Iterator<Int> values):
    int sum=0;
    for each v i nvalues:
        sum += v;
        Emit(term, sum);

        // 최종적으로, ("apple", 4), .... 로 결과가 나올 것이다.
```
### 키워드로 재정리한 mapreduce 과정

1.`Job`. job은 map과 reduce tasks로 나뉘게 될, client가 jobtracker에게 맡길 일을 뜻한다.

Job은 하둡이 자바로 짠 만큼, 이 Job data는 jar 파일과 configuration XML 파일로 이루어져있다. 
- 설정 파일은 configuration.xml로, 실제 data들을 처리하는 내용은 jar파일에 담겨있다.
- running a task의 instance는 a `task attempt`라 부른다.

2.`split`. split은 map과정의 data input 단위이다. HDFS에서 저장돼있던 block과 거의 유사한 사이즈이다.

map()을 수행하는 mapper의 개수는 input size따라 정해진다. 내가 정하는게 아니다!

3.`sort`. mapper노드에서는 mape()수행 후에 local sort와 partitioning까지 처리 후, spill까지 거쳐 로컬 storage에 저장됐던 spill file들을(중간 결과들을) block 단위로 shuffle한다. 즉, block단위로 파일을 copy해서 reducer로 보낸다.

4.`merge`. reducer노드에서는 받아온 block들을 key 기준 sorting 후 reduce()한다.

5.`reduce`. 이 reduce의 개수는 내가 정할 수 있다. partion의 개수만큼 생긴다.

최종적으로 reduce()의 output으로 part라는 단위의 output이 생긴다. 이 part들은 HDFS에 저장된다.


### Map() 이전과 Reduce() 이후의 데이터 처리 작업

사실 map()과 reduce()로 각각 데이터들을 넣어주기 전에, 더 많은 데이터 처리 과정이 있다.

생각해보면 Data가 저장되는 File System은 HDFS이다. 

Map과 Reduce에서 쓰는 파일 형식은 <k,v>의 map 형식이다. 그리고 HDFS는 block 단위의 여러 종류의 파일일 것이다.

서로 다른 두 체계를 건너는 과정에서는 **형식 변환**이 필수일 것이다.

1.mapper에게 input을 넘겨주기 전, 실제론 `InputFormat`이라는 abstract class(하둡 예전 버젼에서는 interface였다고 한다.)가 관여한다. 
- InputSplit: data의 위치(어느 노드인지)와 그 범위(data의 시작과 끝)
- RecordReader: Input split이 정한 범위의 Binary data를 <k,v>로 바꾼다.

이렇게 HDFS에 저장돼있던 각 data들을, 위의 두 개의 객체를 통해 mapper에게 줄 input 형식으로 바꿔주는 것이다.

젬나이에게 물어본 InputFormat의 대략적인 코드는 다음과 같다.

```java
public abstract class InputFormat<K, V> {

    /**
     * 1. 파일을 어떻게 쪼갤지 결정하는 메서드
     * 논리적인 조각(InputSplit)들의 리스트를 반환합니다.
     */
    public abstract List<InputSplit> getSplits(JobContext context) 
        throws IOException, InterruptedException;

    /**
     * 2. 쪼개진 조각을 실제로 한 줄씩 읽는 일꾼(RecordReader)을 생성하는 메서드
     * 여기서 K, V 타입이 결정됩니다.
     */
    public abstract RecordReader<K, V> createRecordReader(InputSplit split, 
                                                         TaskAttemptContext context) 
        throws IOException, InterruptedException;
}
```
두 메서드의 역할 상세

① getSplits(): "피자를 몇 조각으로 나눌까?"

데이터가 어디에(어느 서버에) 있는지 확인하고, 이를 몇 개의 InputSplit으로 나눌지 계산합니다.
- 중요한 점: 여기서 파일이 물리적으로 잘리는 게 아니라, "0번 맵퍼는 A파일의 0~128MB까지 읽어라"라는 **지도(Metadata)**만 만듭니다.
- 이 메서드의 결과에 따라 맵퍼(Mapper)의 개수가 결정됩니다. (Split이 10개면 맵퍼 10대 작동!)

② createRecordReader(): "나눠진 피자를 어떻게 한입씩 먹을까?"

실제로 데이터를 읽어서 사용자님이 맵퍼에서 쓸 수 있는 **$K(Key), V(Value)$**로 변환해주는 도구를 리턴합니다.
- 예를 들어 TextInputFormat이라면, 이 메서드는 **LineRecordReader**라는 일꾼을 반환합니다.
- 이 일꾼은 파일 끝을 만날 때까지 nextKeyValue()를 호출하며 데이터를 뽑아냅니다.

정리하자면, 

1. HDFS상에서 binary로 저장됐던 파일 data를 split으로 나누어 놓는다. 즉, 기존 raw data(binary 파일 혹은 text 파일 등등)를 InputSplit 타입의 데이터 사이즈로 나눠 놓는다.
- 단지 raw data를 크기 순으로, 논리적으로 나누어 놓은거라 실제 읽어들이면서 원하는 타입(mapper의 input으로 줄 수 있는 타입, 즉 <k,v>)을 만드는 건 RecordReader가 한다.
- 여기서 Map task 개수가 결정되는데, 이게 mapper의 개수가 된다!
> 1 InputSplit = 1 Map Task = 1 Mapper
2. RecordReader가 데이터를 읽어들여 <k,v>라는, 임시 key에 data를 mapping한 형태로 mapper에게 보내준다.
- 이때! InputSplit은 단순히 크기 단위로 잘라둔거라, 의미 단위가 아니다. 
> (예: (inputsplit0)"나는 사과를 먹" (inputsplit1)"었다. 나는 포도")
- RecordReader가 다음 block(HDFS상에서는 블록 단위로 저장돼있다.)을 NW를 넘나들며 시작-종료 위치를 보정해 읽어온다.

2.Reducer 이후, 이번에는 OutputFormat이 관여한다.

이번에는 Recordwriter만 존재한다.

```java
public abstract class OutputFormat<K, V> {

    /**
     * 1. 실제 쓰기 작업을 수행하는 RecordWriter를 생성
     */
    public abstract RecordWriter<K, V> getRecordWriter(TaskAttemptContext context) 
        throws IOException, InterruptedException;

    /**
     * 2. 출력 경로 중복 여부 등을 체크 (작업 시작 전 호출)
     */
    public abstract void checkOutputSpecs(JobContext context) 
        throws IOException, InterruptedException;

    /**
     * 3. 작업 성공/실패 시 파일 관리 로직을 담당하는 커미터 반환
     */
    public abstract OutputCommitter getOutputCommitter(TaskAttemptContext context) 
        throws IOException, InterruptedException;
}
```

젬나이가 사용법도 아주 친절히 알려준다.

실무자: RecordWriter는 무엇을 하나요?

RecordReader가 바이트를 객체로 바꿨다면, RecordWriter는 객체를 다시 바이트로 바꿉니다.

```java
write(K key, V value): 리듀서가 context.write(k, v)를 호출할 때마다 실행됩니다.
```

포매팅: 예를 들어 TextOutputFormat의 RecordWriter는 키와 값 사이에 탭(Tab) 문자를 넣고, 끝에 **줄바꿈(\n)**을 붙여서 텍스트 파일로 만듭니다.

압축: 설정에 따라 데이터를 쓰면서 동시에 Gzip이나 Snappy로 압축하기도 합니다.

요는, **RecordWriter가** reduce()의 마지막 줄에서 write(k,v)를 호출할 때마다 ****정해진 타입의 파일로 바꿔준다.**
- **reducer가 내뱉은 결과물 <k,v>를 어떤 형식의 파일로 HDFS에 저장할지 정한대로 반환한다.**

**지금까지 InputFormat과 OutputFormat을 살펴봤는데, 모두 abstract class로서 어떤 파일 포맷을 만들지에 관한 내용이었다.**

하둡에서는 미리 이 InputFormat과 OutputFormat을 각각 상속받은 기본 클래스들이 있다.

InputFormat:
- TextInputFormat
- KeyValueTextInputFormat
- SequenceFileInputFormat

OutputFormat:
- TextOutputFormat
- SequenceFileOutputFormat
