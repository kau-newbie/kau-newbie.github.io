# Hadoop Example - WordCount

앞선 Java나 javac, 그리고 환경변수 설정에 대해서는 과제에서 이미 주어져 뭐라 쓸 수가 없다. 배우고 돌아오겠다.

### Hadoop Mode

하둡에는 세 가지 모드가 있다고 한다.

1. Standalone (or local) mode
- There are no *daemons running and everything runs in a single JVM.
> daemon: NameNode daemon, DataNode daemon, JobTracker daemon, TaskTracker daemon...etc.
- 디버깅용이라고 한다. 네트워크 사용x. 내 로컬 컴퓨터 내에서 돌린다(=single JVM).
2. pseudo-distributed mode
- The Hadoop daemons run on the local machine, thus simluating a cluster on a small scale.
- 내 로컬 컴퓨터 내에서 여러 JVM process를 띄운다고 한다.
    - N/W 사용 O (127.0.0.1, 즉 localhost)
    - daemon들을 상주시킨다.
3. Fully distributed mode
- 실제 클러스터에서 daemon들을 상주시킨다. 실전용.

나는 2번 pseudo-distributed mode로 진행했다.

### WordCount.java

combiner가 없는 버전의 wordbount는 다음과 같다.

```java
package k132.myhadoop.wordcount;
import java.io.IOException;
import java.util.StringTokenizer;
import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.IntWritable;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.Reducer;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.input.TextInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;
import org.apache.hadoop.mapreduce.lib.output.TextOutputFormat;
public class WordCount {
    public static class Map extends Mapper<LongWritable, Text, Text, IntWritable> {
    private final static IntWritable one = new IntWritable(1);
    private Text word = new Text();
    @Override
    protected void map(LongWritable key, Text value, Context context) throws IOException, InterruptedException {
        String line = value.toString();
        StringTokenizer tokenizer = new StringTokenizer(line);
        while (tokenizer.hasMoreTokens()) {
            word.set(tokenizer.nextToken());
            context.write(word, one);
        }
    }
}
public static class Reduce extends Reducer<Text, IntWritable, Text, IntWritable> {
    @Override
    protected void reduce(Text key, Iterable<IntWritable> values, Context context) throws IOException, InterruptedException {
        int sum = 0;
        for (IntWritable val : values) {
            sum += val.get();
        }
        context.write(key, new IntWritable(sum));
    }
}
public static void main(String[] args) throws Exception {
    Configuration conf = new Configuration();
    Job job = Job.getInstance(conf, "wordcount"); // 두 번 째 인자는 별명. 웹 대시보드에서 쉽게 찾을 수 있게 라벨링.
    job.setJarByClass(WordCount.class);
    job.setMapperClass(Map.class);
    job.setReducerClass(Reduce.class);
    job.setOutputKeyClass(Text.class);
    job.setOutputValueClass(IntWritable.class);
    job.setInputFormatClass(TextInputFormat.class);
    job.setOutputFormatClass(TextOutputFormat.class);
    FileInputFormat.addInputPath(job, new Path(args[0]));
    FileOutputFormat.setOutputPath(job, new Path(args[1]));
    System.exit(job.waitForCompletion(true) ? 0 : 1); // job이 끝날 때까지 기다림 - waitForCompletion(). true면 verbose모드라고, 터미널에 
                                                        // 로그를 띄워준다. 정상종료면 true라서 exit(0)이 된다.
}
}

```

여기서 main을 보통 driver라고도 부르는 것 같았다. main에서 전체적인 구조를 잡고, 순서를 설정, 의존성을 주입해준다. 그런 의미에서 driver인가? 

이때, combiner를 쓰고 싶다면, 이 예제 같은 경우 Reducer를 그대로 쓸 수 있다.

이때, input의 output이 각각 Text와 IntWritable인 word와 one인 반면, reduce의 input은 Text와 Iterable<IntWritable>이다. 이 점은 하둡 프레임워크가 해결해준다! 걱정할 필요가 없다.

### 자료형

위의 코드를 보면 나오는 Text와 IntWritable이 있다. 각각 하둡 버전의 String과 Int이다.

물론, 이것들도 각각 기본 하둡 전용 데이터 타입들 Interface를 implements한 것이다.

먼저, Wirtable이다.

#### Writable

A Serializable Object.

- Any key or value type in hte Hadoop Map-Reduce framework implements this IF.

이 IF를 구현하는 클래스들(IntWritable, Text....etc)은 보통 static read(DataInput)이라는 정적 메서드를 구현해야 한다.
- 이 read 메서드는 `new Writable()`로 인스턴스를 하나 만든 뒤, `readFields()` 메서드를 실행, 그 후 이 인스턴스를 반환한다.
- 하둡은 N/W로 클러스터 노드들이 통신하기 때문에, Data는 Bytes 형태의 스트림으로 직렬화된다. 이때, 다시 자바 프로그램(JAR)에서 쓰려면, 
- Byte to Java object로 바꾸는 과정이 필요하다.(역직렬화)
- **이 역직렬화를 read메서드에서 instance를 만든 뒤, readFields()로 담아준다.**

하둡에서 기본적으로 제공하는, 이 Writable IF를 implements해놓은 primitives(자바의 원시타입과 1대1 대응되는 타입들)는 다음과 같은 것들이 있다.
> BooleanWritable, ByteWritable, IntWritable, FloatWritable

그 외 primitive는 아니지만 하둡에서 기본 제공하는 타입으로는,
> Text, BytesWritable 등이 있다.

예시 코드는 아래와 같다.

```java

public class MyWritable implements Writable{
    private int counter;
    private long timestamp;

    public void write(DataOutput out)throws IOException{
        out.writeInt(counter);
        ...
    }

    public void readFields(DataInput in)throws IOException{
        coutner=in.readInt();
        timestamp=in.readLong();
    }

    public static MyWritable read(DataInput in) throws IOException{ // self 객체를 반환한다. read메서드!
        MyWritable w=new MyWritable() ;
        w.readFields(in);                 //정의해둔 readField로 역직렬화
        return w;                         //역직렬화한 데이터를 self에 담아서 반환
    }
}

```

### Data의 흐름

따로 설정해주지 않아 자연스럽게 하둡이 처리했지만, map과 reduce외에도 데이터는 많은 처리 과정을 거친다.

#### Reading Data

**InputFormat**

1. data 경로를 정해주면, InputFormat이 경로 내 파일들을 모두 가져와서, HDFS 블록 크기 등을 참고, 몇 개의 InputSplit으로 쪼갤지 정한다.
- input을 분석해서, 'input-specification'을 만든다.
- 예: 0~128MB -> 1번, 128~256MB -> 2번 InputSplit. 즉 InputSplit은 HDFS 상의 블록(physical)을 logical하게 다른 단위로 쪼개놓은 것이다.
2. 이때, InputSplit은 어디서부터 어디까지 읽을 지에 대한 메타 데이터에 불과하다.
3. InputFormat은 일종의 RecordReader를 만드는 factory.
- RecordReader가 InputSplit을 읽어들인 뒤, <K, V>형태로 바꾼다.
- 모든 종류의 InputFormat은 각자의 RecordReader 구현체를 갖는다.
    - 예: LienRecordReader, KeyvalueRecordReader ...etc.
4. RecordReader를 구현해서 input records를 <k,v>로 만들어 Mapper에게 건넨다.

역시나 InputFormat자체는 IF이고, 이걸 상속받은 클래스는 FileInputFormat이다.
- 이때, FileInputFormat은 경로상 모든 파일을 읽어 들인다.
- 별도의 필터링 메서드를 만들 수 있다.
- InputSplit으로 쪼개는 사이즈는 보통 chunk 사이즈이다.
    - 즉, RecordReaders는 chunk들에 관한 메타데이터를 받는다.(file, offset, length of chunk)

마찬가지로 하둡에서 미리 제공하는 FIleInputFormat 구현체들이 많다.
- TextInputFormat, KeyValueTextInputFormat...etc.
> 바로 위의 이 두 예시는 '\n'단위로 텍스트 파일을 읽어 value로 바꾼다.(line마다 value로.)

파일 형태가 아닌 HDFS 상 데이터들을 위한 클래스들도 있따.
- ComposableInputFormat<k,v>, DBInputFormat<T>....etc.

교재에 예시 코드가 나와있다.

`InputFormat`이란 IF가 있고, 얘를 구현한 대표적 클래스가

### Map-Reduce의 최적화(hadoop 1.0)

하둡은 기본적으로 분산 cluster에서의 데이터 처리(Map-Reduce)를 가정하고 있다.

Scalable하게 하둡 알고리즘을 짜려면,

- Avoid Object Creation
> map()안 object하나를 instance할 때, 만일 input이 1억 개면 객체도 1억 번 생성된다.
> - garbage collection을 실행할 때에도 부하가 심하다. 1억 번 해야한다니.

- Avoid Buffering
> hadoop의 각 task(JVM)가 쓸 수 있는 heap size는 제한적이다. 데이터 set 사이즈가 클수록 heap 공간이 부족할 것이다.

하지만 그렇다고 해도 disk에 쓰고 읽어오는 시간과, N/W로 보내는 과정(shuffle)이 너무 병목이 심하다.
- 교재에서는 'Importance of Local Aggregation'이라고 했다.
- 왜냐하면 분산환경에서 각 노드가 늘어남에 따라 계산 효율이 linear 하게 scaling되지 않는데, 그 이유가 노드 간 Communication 때문이라고 한다.
    - 서로 동기화(Synchronization)을 맞춰야 하기 때문이다.
    - 동기화는 읽기/쓰기에 대한 lock일 수도 있고, 파이프라인에 의한 앞 process를 기다리는 일일 수도 있다.

이 모든 걸 고려해 나온 해법은 다음과 같다.
- communication을 피한다. 
> - reducer에게 보내기 전 최대한 많이 작업한다. -> N/W로 전송하는 data 양을 줄인다.
> - combiner를 쓰거나, In-Mapper Combining pattern이 되겠다.

아래는 그 구현이다.(algorithm)

1.`In-Mapper Combining`
- combiner가 아닌, 개발자가 직접 memory위에서 전역변수로 state를 보존하는 것.
> state는 별 거 아니고, 평균 구하는 map-reduce를 예로 들면, 이때의 accumulation sum이다. 합산을 저장할 전역 변수를 계속 메모리 위에 살려두는 것.

2.`Order Inversion`
> 예를 들어, "a 뒤에 b가 등장할 확률 $P(b|a)=\frac{(a,b)가 나온 횟수}{a가 나온 횟수}$ 을 구할 때,
>
> reducer가 다 받아놓고 (a,*)의 개수를 세서 (a,b)가 나온 개수에 나눠주는게 X.
> - 대신에 state를 보존해서 미리 map에서 (a,*)의 개수를 합산 후, 제일 먼저 shuffle로 보내도록 해준다.
> - 제일 먼저 해당하는 reducer에게 전송되도록 하고, reducer에서는 이 state로 바로 나누면 되도록 한다.

먼저, 1번의 `In-Mapper Combining`을  WordCount의 Pseudo Code 예시로 살펴보자.

우선, 기본적인 base code이다.

```java

class Mapper
    method MAP(docid a, doc d)
        for all term t ∈ doc d do
            EMIT(term t, count 1)
    
class Reducer
    method REDUCE(term t, counts[c1,c2,...])
        sum <- 0
        for all count c ∈ counts[c1,c2,...] do
            sum <- sum + c
        EMIT(term t, count s)

```

핵심은 reducer에게 보내기 전, 데이터 자체를 줄여 전송량을 줄이는 것이다.

문서 하나 당, 즉 doc d 하나 당, *해시 테이블*을 만들어 '단어 - sum'을 계산한 후, reducer에게 이 해시테이블을 보내는 것이다.

```java

class Mapper
    method MAP(docid a, doc d)
        H <- new AssociativeArray
        for all term t ∈ doc d do
            H{t} <- H{t} +1
        for all term t ∈ H do
            EMIT(term t, count H{t})

```

많이 줄긴 했지만, 문제는 MAP()이 doc d, 즉 문서 하나에대한 연산이란 것이다. 
- 문서 하나에 대한 여러 word들을 한데 모아서 보내긴 하지만, 여전히 문서는 수천 장~ 그 이상일 것이고,
- 그렇게 되면 아직 중복되는 word에 대한 count들은 많을 것이다.(적어도 수천 개~)

또, 이렇게 MAP은 수천 번 Hash의 생성과 소멸을 반복해줄 것이다. 낭비다!

얘네를 다시 중간에서 reducer에게 보내기 전, 모아주느라 combiner가 필요하단 것이다.

다시 한 번 더 바꿔본다. 이번에는 hash의 재생성과 재소멸(?)을 막기위해 stage를 3개로 나눠준다(초기화->map->정리(cleanup))

```java

class Mapper
    method INITIALIZE
        H <- new AssociativeArray
    method MAP(docid a, doc d)
        for all term t ∈ doc d do
            H{t} <- H{t} +1
    method CLOSE
        for all term t ∈ H do
            EMIT(term t, count H{t})

```

즉, class 내부 state(해시 테이블)를 보존해서 모든 Map()호출마다 누적 후, 마지막에야 합산이 기록된 해시 테이블을 보낸다.
- accumulation

따라서 드디어 combiner가 필요 없다!

위의 디자인 패턴(알고리즘 방식)을 *In-Mapper Combining*이라고 한다.
- state라는 어떤 정보(값)을 계속 mapper 메모리에 보존하면서 여러 번 map의 결과들을 state로 저장 후, 마지막에 reducer에게 보낸다.

In Memory라서 spill하는 combiner와 달리 빠른 이점이 있다.

단, state 값에 의존성이 있을 경우, 순서가 꼬여 race-condition이 발생할 수도 있고,

combiner와 달리 하둡이 자동으로 알아서 spill, 즉 map 결과물들이 쌓이다 메모리가 부족해지면 알아서 EMIT()해주지 않기 때문에 메모리 관리를 직접 해야 한다.

이번에는 wordcount말고 평균을 구하는 예제를 본다.

base pseudo code부터 본다.

```java

class Mapper
    method MAP(string t, integer r)
            EMIT(string t, integer r)
    
class Reducer
    method REDUCE(string t, integers[r1,r2,...])
        sum <- 0
        cnt <- 0
        for all integer r ∈ integers[r1,r2,...] do
            sum <- sum + r
            cnt <- cnt + 1
        ravg <- sum/cnt
        EMIT(string t, integer ravg)

```

특별할 건 없고, 이대로 combiner를 추가하려면, 이름만 class Combiner에 코드는 갖다붙이면 될 것이다. (구현은 제일 위의 코드처럼 setCombinerClass()로 해줄 것이다.)

하지만 문제가 있다. 여기서는 reduce 코드의 output과 reduce코드의 input이 일치하지 않는다!

따라서 Reducer를 그대로 쓰지 않고, combiner는 합산만(accumulation) 해 놓은 *Pair(sum, cnt)*를 두 번째 인자로 출력해야 겠다.

여기서 또 신경쓸 게 있는데, combiner는 옵션, 즉 있으나 없으나 최종 결과를 해치면 안된다.

mapper가 combiner없이 바로 output을 reducer에게 준다면, 분명 오류가 날 것이므로, mapper의 출력 또한 combiner의 출력처럼 *Pair(sum, cnt)*형태여야 한다.

```java

class Mapper
    method MAP(string t, integer r)
            EMIT(string t, pair(r,1))

class Combiner
    method COMBINE(string t, pairs[(s1,c1),(s2,c2),...])
        sum <- 0
        cnt <- 0
        for all pair(s,c) ∈ pairs[(s1,c1),(s2,c2),...] do
            sum <- sum + s
            cnt <- cnt + c
        EMIT(string t, pair(sum,cnt))

class Reducer
    method REDUCE(string t, pairs[(s1,c1),(s2,c2),...])
        sum <- 0
        cnt <- 0
        for all pair(s,c) ∈ pairs[(s1,c1),(s2,c2),...] do
            sum <- sum + s
            cnt <- cnt + c
        ravg <- sum/cnt
        EMIT(string t, pair(ravg,cnt))

```

문제 없이 동작하는 모습이다.

여기서 한 발 더 나아가서, In-Mapper Combiner 방식을 적용해보자.

우선, combiner를 없애고,

```java

class Mapper
    method INITIALIZE
        S <- new AssociativeArray // key -value 쌍 list을 의미한다.
        C <- new AssociativeARRAY
    method MAP(string t, integer r)
        S{t} <- S{t} + r
        C{t} <- C{t} + 1
    method CLOSE
        for all term t ∈ S do
            EMIT(string t, pair(S{t},C{t}))

```

로 Mapper만 고쳐주면 된다.

그리고 2번으로 넘어가기 전, 잠시 짚고 넘어가야 할 두 가지 방법이 있다.

예를 들어, 검색 엔진 회사에서 연관 검색어를 위한 거대한 NXN 테이블(N: vocabulary size. 알고 있는/알려진 단어 수)에 특정 단어 셋 (a, b)이 몇 번 등장했는지 기록한다고 해보자.
- 의미론적 유사도 측정 얘기가 나온다...(measuring semantic distance)

아무튼 정말 거대한 데이터 사이즈가 나올 것은 자명하다. 우리는 이 문제를 map-reduce로 해결해보려 한다.

이런 상황에서 가장 기본적인 접근은 다음과 같다.
- Mapper generate partial counts
- Reducers aggregate partial counts

어차피 map-reduce를 여러 번 반복해야 할 것이기 때문이다.

아무튼 이렇게 partial counts를 하려면 매번 더 효율적으로 하는 법을 찾아야 한다.

이때! 이미 두 가지 방법이 알려져있다고 한다.

#### Pairs

`Each mapper takes a sentence:` 
- Generate all co-occurring term pairs
- For all pairs, emit (a, b) → count

1. 각 mapper는 문장을 입력으로 받아서, 모든 단어쌍을 만들고, 단어쌍을 key로 한다. (key space가 생성되겠다!)
2. 각 단어 쌍에 대한 빈도 수 (count)를 1로 붙여서 (즉 pair쌍이 나올 떄마다) value로 삼고, 이 한 k-v 쌍을 즉시 emit() 해준다.

`Reducers sum up counts associated with these pairs`

reducer들은 각자 받은 이 pair에 대해서 sum up 한다. 

`Use combiners!`

combiner를 써주는 게 아무래도 효율적일 것이다. (매 pair 마다 N/W로 shuffle해주면 부하가 심할 것이다.)


Pseudo Code는 아래와 같다.

```java

class Mapper
    method MAP(docid a, doc d)
        for all term w ∈ doc d do
            for all term u ∈ NEIGHBORS(w) do
                EMIT(pair(w,u), count 1)

class Reducer
    method REDUCE(pair p, counts[c1,c2,...])
        sum <- 0
        for all count c ∈ counts[c1,c2,...] do
            sum <- sum + s
        EMIT(pair p, count s)

```

장점:

- 구현이 쉽고, 이해하기도 쉽다.

단점:

- 여전히 N/W로 보낼 데이터 - Pair가 많다. (too many to sort and suffle )
- combiner가 할 일이 별로 없다. 즉, 효율이 떨어진다.

다음 방법을 살펴보자.

#### Stripes

pair들을 associative array로 그룹 짓는다.

```

예:

(a,b) -> 1
(a,c) -> 3
(a,d) -> 5
(a,e) -> 2

위의 자료를 다음과 같이 바꾼다.

a -> {b:1, c:3, d:5, e:2}


```

각 mapper는 한 문장씩 받아서 term pairs를 만든다.

각 term 끼리의 pair는 a -> {b: cntb, c: cntc, .... } 형태일 것이다. 이걸 mapper에서 emit() 한다.

reducer에서는 이대로 덧셈을 수행하면 된다.

```java

class Mapper
    method MAP(docid a, doc d)
        for all term w ∈ doc d do
            H <- new AssociativeArray
            for all term u ∈ NEIGHBORS(w) do
                H{u} <- H{u} + 1
            EMIT(term w, Stripe H)

class Reducer
    method REDUCE(pair p, stripes[H1,H2,...])
        Hf <- new AssociativeArray
        for all stripe H ∈ stripes[H1,H2,...] do
            Sum(Hf, H)
        EMIT(term w, Stripe Hf)

```

장점:

- sorting이 훨씬 적고, shufflign of  key-value pairs도 훨씬 적다. 데이터 자체가 줄었기 때문.
- Can make better use of combiners.
- pairs는 n개 중 2개 선택이라 n*(n-1) 즉, O(n^2)이지만, stripes는 단어(term) 당 딱 한 개, 즉 O(n)이다.

단점:

- 구현이 까다롭다.
- 단순 pair<,>에서 해시맵 object라 무거워졌다.
- 근본적인 단어들의 총 개수가 변하진 않는다.


그리고 2번, `Order inversion`이다.

일단 이 문제의 예시는 Relative Frequencies를 찾는 문제로 시작한다.

지금까진 단순히 counting만 (++, --) 했다면, 지금은 A라는 키워드에 대해 어떤 키워드(단어)와 같이 나온 횟수가 더 많은지 알아보고자 한다.

$$f(B|A)= \frac{N(A,B)}{N(A)} = \frac{N(A,B)}{\sum_{B'} N(A,B')}$$

1.Stripes로 풀어보기

a -> {b: cntb, c: cntc, .... } 하나에 a와 찾고자하는 특정 단어 b간의 빈도 수와, 그 외 모든 경우의 수가 포함됐기에,

**One pass to compute (a,*)이다.** 즉, 이 데이터 하나만 전송받아서, 바로 계산하면 된다.

2.Pairs로 풀어보기

상대적인 빈도를 측정하기 위해선, 모든 경우의 빈도 수 합산을 알아야 한다.

이건 순서에 의존성이 생긴다. 즉, 모든 경우를 보기 전까진 모든 경우의 총 개수를 알 수 없다.

단, 여기서 Buffering은 사용하지 않는다. 데이터가 많아지면 넘칠 우려가 있기 때문이다.

그렇다면, 우리는 여기서 reducer에게 이 총합, 즉 수식의 분모에 들어갈 값을 가장 먼저 보내놓고, 나중에 reducer가 나눌 때 쓰도록 하는 방법을 찾아본다.

- 항상 (a,*)가 제일 먼저 리듀서에 들어가도록 한다. 애초에 key 기준 정렬시, 가장 앞에 가게 한다.
    - 예를 들어, reducer는 이 (a,*) key에 대한 value를 가장 먼저 받고, shuffling으로 넘겨받은 찾고자 하는 빈도 값 (a,b)에 대한 value에 나눠주기만 하면 바로 찾고자 했던 $f(B|A)= \frac{N(A,B)}{N(A)}$ 가 구해진다
- 단, partitioner를 잘 써서 이 경우에는 모든 a에 대한 쌍의 경우는 같은 reducer에게 가도록 한다.
- reducer에서는 특정 key-value pairs에 대해 state를 잘 유지하고 있어야 한다.
    - 여기서 말하는 건 분모로 들어가야 할 (a,*)이다.

강의에서는 이 시점에서 Pairs와 Stripes를 `Synchronization` 측면에서 비교했다.

여기서 synchronization이란, 제미나이의 정의를 빌리자면,

```

맵리듀스는 여러 컴퓨터가 쪼개서 일하기 때문에, Reducer가 나눗셈이나 평균 같은 복합 연산을 하려면 "필요한 데이터 조각들이 한 컴퓨터(Reducer)에, 그것도 올바른 순서대로" 모여야 합니다. 이를 데이터의 동기화 문제라고 부릅니다.

```
라고 한다.

즉, 이 예시에서는 분모(총 빈도 수)와 분자(특정 빈도 수)를 따로 받아와 마무리 계산을 해야 하는 상황이다.

(제미나이 보고 요약을 시켰다.)

##### 1. Approach 1: "정렬(순서)의 마법으로 해결하자"  **Pairs 패턴**

> "데이터 구조는 단순하게 유지하되, 하둡의 정렬(Sort) 기능을 극한으로 조작해서 해결하자."

* Sort keys...: `(a, *)`가 `(a, b1)`보다 무조건 먼저 오도록 하둡의 기본 정렬 순서를 커스텀하게 바꿉니다 (Order Inversion 기법).
* Partition key space...**: 같은 부모 `a`를 가진 모든 데이터 쌍들이 반드시 동일한 Reducer로 흘러 들어가도록 파티셔너(Partitioner)를 조작합니다.
* Hold state...: 아까 질문하셨던 내용입니다! 먼저 도착한 `(a, *)`의 합계 값(32)을 Reducer 메모리에 잘 저장(`Hold state`)해 두고, 뒤이어 들어오는 자식 Key 조각들과 결합해 연산합니다.

##### 2. Approach 2: "자료구조의 마법으로 해결하자"  **Stripes 패턴**

> "순서 맞추는 건 너무 복잡하고 피곤하다. Mapper가 데이터를 던질 때부터 연산에 필요한 조각들을 연상 배열(Map/자료구조)에 한 번에 묶어서 던지자."

* construct data structures...: 데이터를 하나씩 `(a, b1)`, `(a, b2)` 따로 던지지 않고, `a -> {b1: 3, b2: 12, b3: 7}` 처럼 하나의 거대한 주머니(Map 자료구조)를 만들어서 통째로 던집니다.
* Each reducer receives all the data...: 이렇게 하면 Reducer는 이 주머니 하나만 딱 열어보면 그 안에 분자들($3, 12, 7$)도 들어있고, 그것들을 싹 더하면 분모($32$)도 바로 구할 수 있습니다. 순서를 조작하거나 상태를 메모리에 기억해 둘 필요가 전혀 없어집니다.

##### 한 줄 요약

> *Pairs(접근법 1)*는 하둡의 *정렬과 분산 순서(`Ordering`)*를 정교하게 컨트롤해서 데이터를 동기화하는 방식이고, *Stripes(접근법 2)*는 애초에 연산에 필요한 데이터를 *연상 배열(`Data Structure`)*이라는 하나의 덩어리**로 묶어서 Reducer에게 한방에 배달하는 방식입니다.


이외에도 'secondary sorting'이라고, (k,v)가 있을 때, v 기준 sorting에 대한 방법 두 가지 정도를 비교하는 내용도 있었다.

1. Buffer values in memory, then sort
> 문제: 이미 디스크에 i/o 해가며 sorting 했는데, 자바코드로 별도로 또 정렬? 낭비다.

2. 하둡 프레임 워크를 이용할 건데, 그러려면 key 기준 sorting하니까 키에 (k,v)를 넣고, value자리에 의미 없는 값을 넣겠다.
- composite intermediate key, 즉 복합키라고 한다.

#### 정리하자면, Synchronization이 특정 목적의 계산을 위해선 중요한대, 

- stripes 같이 영리하게 data structure를 통해 해결하거나,
- key sorting을 이용해 (보내는) 순서를 잘 조절하거나
- partitioner 조작을 통해 어느 reducer가 어느 key에 대해 처리할 건지 잘 지정하거나
- state를 잘 저장한다. (In-Mapper Combiner 패턴)

#### 마지막으로 생각해볼 점들(TradeOffs)

(역시 재미나이 딸깍을 써봤다.)

빅데이터 시스템을 설계할 때는 무조건 좋은 기술이란 없으며, **데이터의 개수, 크기, 메모리 상황**에 따라 장단점을 잘 저울질(Tradeoff)해야 한다는 본질을 3가지 축으로 설명하고 있습니다.

##### 1. Number of key-value pairs (데이터 '개수'의 문제)

데이터 조각들의 **개수가 너무 많아질 때** 발생하는 비용(단점)입니다. 주로 **Pairs 패턴**이 이 문제에 걸립니다.

* **Object creation overhead:** 자바에서 `(a, b1)`, `(a, b2)` 같은 Key-Value 쌍 객체를 수억, 수십억 개씩 새로 만들면(new), 컴퓨터 메모리에 과부하가 걸리고 자바의 쓰레기 수거기(Garbage Collector)가 비명을 지르게 됩니다.
* **Time for sorting and shuffling...**: 개수가 많으면 셔플 단계에서 컴퓨터들끼리 데이터를 정렬하고 네트워크로 주고받는 데 엄청난 시간이 걸립니다.

##### 2. Size of each key-value pair (데이터 '크기'의 문제)

반대로 데이터 한 조각의 **크기가 너무 커질 때** 발생하는 비용입니다. 주로 **Stripes 패턴**이 이 문제에 걸립니다.

* **De/serialization overhead:** Stripes 패턴은 `a -> {b1:3, b2:12, ...}` 처럼 거대한 Map 자료구조를 통째로 던집니다. 이 거대한 주머니 객체를 네트워크로 보내기 위해 바이트 바이트 단위로 쪼개고(직렬화, Serialization), 받을 때 다시 객체로 조립하는(역직렬화, Deserialization) 과정에서 **CPU 연산 비용**이 엄청나게 소모됩니다.

##### 3. Local aggregation (디스크로 보낼지, 메모리에서 끝낼지)

데이터를 다른 컴퓨터로 보내기 전에 Mapper 컴퓨터 안에서 미리 뭉치는 작업(Local aggregation)에 대한 장단점 비교입니다.

* **Opportunities... varies:** 데이터가 어떻게 들어오느냐에 따라 미리 뭉쳐서 이득을 볼 수 있는 기회의 크기가 다릅니다. (중복이 많은 데이터면 대박이고, 중복이 없으면 뭉쳐봤자 효과가 없겠죠.)
* **Combiners vs. in-mapper combining:** * 하둡의 기본 기능인 **Combiner**를 쓰면 안전하지만 디스크를 거쳐야 하므로 조금 느립니다.
* 반면, 우리가 배운 **In-mapper combining**은 100% **RAM(메모리)** 안에서 처리하므로 압도적으로 빠르지만, 만약 단어 종류가 너무 많으면 RAM 용량이 부족해 컴퓨터가 뻗어버릴 수 있는 위험(Out of Memory)이 있습니다.


* **RAM vs. disk vs. network:** 결국 빅데이터 엔지니어는 **"내 시스템의 RAM을 더 쓸 것인가(In-mapper), 안전하게 디스크를 더 쓸 것인가(Combiner), 아니면 그냥 가공 없이 네트워크로 다 던져버릴 것인가"** 사이에서 최적의 밸런스를 찾아야 한다는 뜻입니다.

##### 한 줄 요약

> **"Pairs는 데이터 개수가 많아져서 네트워크와 정렬이 느려지는 게 단점이고, Stripes는 덩어리가 커서 CPU 직렬화가 느린 게 단점이다. 이를 해결하기 위해 메모리(RAM)를 써서 미리 뭉칠 것인지(In-mapper), 디스크를 쓸 것인지(Combiner) 데이터 상황에 맞게 밸런스 겜을 잘해야 한다"**
