# Hadoop 공부 정리

처음 Google이 **Google File System(GFS)**를 논문을 통해 발표한 뒤로,

하둡에서는 **Hadoop Distributed File System(HDFS)**로 구현했다.

둘 다 분산 저장 파일 시스템이므로, 몇 가지 가정들이 내포돼있다.

GFS는 다음과 같은 가정을 하고 있다.

1. 각 물리적인 노드들은 고스펙의 하드웨어가 아닌, 저렴한 하드웨어로 구성됐다.
- scale "out"을 강조한다. (not scale up)
2. 저렴한 하드웨어니까 고장이 더 자주 날 것이다.
- 감안하고 시스템을 설계했다는 얘기이다.
- 즉, 대처를 잘 하도록 (Fault tolerance) 설계했다.
3. 일반적으로 아주 큰 파일들을 처리한다.
- 과거에는 작은 크기의 많은 파일들이었다면, 오늘날(빅데이터)은 파일 하나하나의 사이즈가 어마어마하게 크다고 한다.
- 딱 듣기에도 한 번에 크게 디스크에서 -> 메모리로 읽어들이면 유리할 것 같다.
    - 아무래도 같은 파일 하나 안에 밀접한 데이터들이 모여있을 확률이 높기 때문이다.
4. 파일은 한 번만 쓰고, 수정 x, 뒤에 덧붙인다.(append)
- streaming data를 생각하면 이해가 쉬웠다. log data 같은 경우, 쏟아져 나올 뿐만 아니라, 수정을 요구하지 않는다. 주로 읽을 뿐이다.
5. large streaming reads over random access
- 4번과 일맥상통하는 것 같다. 크게 한 번 sequential하게 읽어서 메모리로 들인 뒤에, 빠르게 처리하는 게 좋다. (되돌아갈 여유가 없다! 하도 많아서.)
- random access는 HDD나 SDD나 느릴 뿐더러, 수정하려 드는 순간, 분산 환경에서는 race condition을 유발한다.

따라서 GFS는 다음과 같은 디자인이다.

1. File은 chunk 단위로 고정됐다. 
- 당시 64MB였다고 한다.
2. 3+ chunkservers에 replica를 저장해둔다.
- 데이터 노드 하나가 오류로 고장나도, 나머지 replica에서 해당 data를 다시 구해올 수 있다.
3. masternode 하나가 metadata(노드들 안 데이터 위치)를 관리한다.
- 중앙 집중식이라 관리가 몹시 편하다. 단점도 물론 있겠다.
4. no data caching
- 위에서 data는 수정할 필요없이 그저 한 번에 읽는다고 했다.
- 즉, 재사용할 필요가 없으니 caching을 하지 않는다.
5. simplify the API
- push some of the issues onto the client
- 실제 file system에서는 여러 복잡한 문제들을 처리해준다고 한다. 중복을 처리하거나, 하는 식인데, 이런 복잡한 문제들을 master node에서 처리해주지 않고, 그저 client 단에 맡긴다고 한다.
- 이 부분은 정확히 실제 file system에 대한 공부가 필요한 것 같다.

## HDFS

### HDFS의 목적

1. 왕 큰 분산 File System
- 10K nodes, 100 million files(1억 F), 10PB
2. 저렴한 하드웨어 사용
- GFS의 클론답게, 마찬가지로 저렴한 하드웨어를 사용을 가정한다.
- file들은 replica로 저장하고 있다. hw failure에 대응해야 하기 때문이다.
3. Optimized Batch Processing
- data가 있는 노드로 map 같은 프로세스들이 이동해서 실행된다. 한꺼번에 많은 data를 다룰 때 유리하다.
- batch processing이란 방대한 일련의 일들을 시키는 것인데, 이는 빠른 응답에 맞지 않는다.
- 다만, 한꺼번에 많은 양의 일을 수행할 순 있다. (throughput >>> latency)
- 젬나이가 좋은 비유를 해줬는데, '빨리 도착하기'가 아닌, **'한꺼번에 옮기기'**란다.
4. User Space, runs on heterogeneous OS
- User space 단에서 실행되기 때문에, 즉 kernel space의 file system을 사용하지 않기 때문에
- OS 종류가 요상해도 어디서든 다 돌아간다.
- 요컨데 추상화이다! 다만, 진짜 다 되는지는 의문이 든다.

### HDFS의 구성

HDFS는 다음과 같이 구성돼있다.

- Application(HDFS client)
- HDFS namenode
- HDFS datanode

**HDFS Client**

namenode에게 file name (, block id)을 통해 block의 위치를 받아온다.
> data node list for each block (file을 block 단위로 쪼개어 저장하고 있다.)
> - "client picks first data node for each block"이란 거 보니, 가까운 순인가..?
>   - 그렇다고 한다. local Nodes first라고 언급됐다.
> - 와중에 또 block들을 sequentially하게 읽는다고 한다. 아마 요청한 파일에 대해, 넘겨준 list 속 block들을 원본 파일에 맞는 순서대로 배치해서 주었나보다.

그 후, 해당 위치(노드)로 가서 block data를 받아온다.

client가 `intelligent` 하다는데, local F sys 에서는 보통 client가 하는 일이 없다.
> 보통 하드 디스크 속 위치나, 네트워크 상의 (읽고자 하는 파일이 있는)서버 위치 같은 건 모르고, 그저 파일을 읽고 쓰기를 OS에 요청한다.

하지만 HDFS에서는 client가 직접 namenode에게 file 위치를 물어보고, 

metadata를 받고나서는 해당 F이 들어있는(replica) 많은 노드 중에서도, 제일 가까운 노드로 찾아간다. 

아래와 같은 순서로 찾는다.

1. 같은 노드(서버)
2. 같은 Rack (서버 컴퓨터 여러 개 끼워놓은 그거.)
3. 그 외

직접 찾아가서 data를 요청한다.

data를 받은 후엔 checksum(CRC32)을 이용해 파일의 유효성을 검증한다. 512B마다 파일을 검증하는데, 

datanode에 있던 checksum과 대조해 검증한다. 실패하면 다른 replica를 가져온다.

**HDFS namenode**

1\. meta data

DFS(Distributed File System)에서는 보통 클러스터(노드 그룹) 안에 하나만 있다. (단순한 중앙집중식 구조)
- HDFS는 여기에 더해, 실시간 동기화는 아니지만 secondary name node를 둔다.

HDFS의 **계층구조와 권한 정보, 그리고 block 들의 실제 위치 등**을 FSDirectory, INode, BlockInfo 등에 담고 있다.

```
INodeDirectory: 하위에 다른 INode들을 리스트로 가집니다.

INodeFile: 파일의 실제 정보를 담고 있으며, 가장 중요한 블록 목록을 가집니다.
```
meta data를 갖고 있다고 볼 수 있는 것이다.
- meta data를 memory에 저장하고 있다.
    - list of files
    - list of blocks for each file
    - list of datanodes for each block
    - file attributes (생성 시간, replication factor: 1이 default고, replica 추가마다 +1 등등)
- 전원이 꺼지면 잃어버리기 때문에, 주기적으로 datanode로부터 heartbeat를 통해 활성상태 정보를 받아온다.
- transaction Log도 가지고 있다. namenode는 기본적으로 하나뿐이라 비정상 종료시 어떻게든 복구시키기 위한..., 최소한의 노력이다.

요컨데, Client의 요청에 "이 블록은 어느 파일 안에 있고, 이 파일은 어느 노드 안에 있어."하고 알려준다.

2\. Coordinating file operations

하나의 client가 이미 write 작업 중인데, 다른 client가 같은 블럭 id에 대한 요청을 보내면 대기시킨다.(응답을 미룬다.)
- lock을 건다.

3/. health 관리

data node들로부터 주기적으로 heartbeat를 받는다. (그냥 ping을 받는다.)

새로 노드가 들어오거나, 원래 있던 노드가 죽었으면 block re-replication and rebalancing이 일어난다.

**HDFS datanode**

각 노드들은 특별할 것 없이, 그냥 local한 linux file system(혹은 그 외 File system)을 쓴다.
> 파일은 보통 블록(block) 단위로 쪼갠 뒤 저장하는데,
>
> GFS에서는 64MB (00년대 초 나온 논문에 따르면ㅎㅎ) 사이즈이고, typically 128MB 사이즈라고 한다.

block에 대한 meta data, 이를테면 CRC 같은 에러 체크용 코드를 둬서 저장된 block을 관리한다.

또, client가 요청하면 data를 넘겨준다.

주기적으로 name node에 가진 block들을 알려준다.(ping)
- every 10th heartbeat is a block report라고 한다. 
- heartbeat를 보내지 않는 노드가 발생하면? 저 아래에 나온다.

### Block Placement

client가 'name node로부터 받아온 meta data'를 참고하여, block 위치를 찾는 순서가 있었다.

block을 쓴 뒤 replica를 만들 때는 그 순서가 사뭇 다르다. (client는 가장 가까운 노드부터 읽었다.)

1. on local node
- 클라이언트가 돌아가고 있는 그 컴퓨터(노드, 서버)에 저장한다.
2. on a remote rack
- 그 다음 replica는 멀리 떨어진 replica이다
- 이건 aws에서 AZ라고 불렀다. 한 노드가, 한 랙이 고장나면 데이터가 절멸이기 때문이다. 어떻게든 물리적으로, 거리상으로 분산해서 저장해야 한다.
3. on same remote rack
- 같은 랙 안에 위치한 노드에 replica를 만든다.
4. randomly placed
- 그 다음 부터 (4번째 부터) 랜덤한 노드에 replica를 만든다.


### Data Pipelining

client가 file에 write 연산을 수행할 때, 즉 file에 무언가를 쓸 때 특이한 과정을 거친다.

1. client는 특정 file에 write 작업을 하고 싶다. 그 file은 block0, block1, block2로 이루어졌다고 가정하자.

2. 먼저 위에서 잠깐 언급했듯이, name node에게 block0에 write를 하고 싶다는 요청을 보낸다.

3. 다른 노드에서 write 하고 있지 않다면, namenode는 client에게 해당 block0의 replica들이 있는 노드들을 알려준다.

4. pipe lining으로 써야할 block0이 있는 노드들과 client가 일종의 연결을 이룬다. client는 block0을 특정 노드들 중 하나에 쓰고,

5. 그 노드는 rack을 잇는 NW 선을 따라 (switch의 routing으로 길을 찾아간다.) 다른 나머지 노드로 찾아가 똑같이 수정하고,

6. 마지막으로 남았던 이 나머지 노드는 다 쓴 후, 역순으로 client에게 쓰기 완료 신호를 보낸다.

7. client는 namenode에게 쓰기 완료신호를 보내고, 다음 block1쓰기를 요청한다.
- 이때, 각 노드들은 새로운 쓰기 사실을 namenode로 알리게 될 것이다. 이건 hearbeat와 별개라고 한다.

### Rebalancer

aws의 로드 밸런서처럼, 노드 별로 가용률이 다를 때 Rebalancer가 작동한다.

예를 들자면, 하나의 노드는 60%를 사용 중인데, 어떤 노드는 70%나 disk를 사용중이다.

이때 보통 threshold를 정해서 해당 %만큼 차이가 난다면, rebalancer가 작동한다.
- 그렇다고 항상 rebalancer가 작동줄일 수는 없는게, NW는 항상 HDFS의 bottleneck으로 존재하고 있기 때문에, 
- rebalncing을 위해 rebalancer가 바쁘게 움직일수록, NW는 traffic이 과도하게 몰릴 것이다.
- 해서, 보통 하루에 몇 % 이하만(NW throughput) 쓸 수 있도록 정해준다.

하지만 보통 balancing은 워낙 잘 돼서 (hadoop이 그 정도라고 한다.) 새로 노드가 추가될 때 주로 작동한다고 한다.

### Re-replicating missing replicas

앞에서 datanode들은 namenode에게 주기적으로 heartbeat를 보낸다고 했다. 일종의 생존신고이다.

그렇다면 heartbeat를 더이상 보내지 못한다면? 예컨데 서버가 다운됐다면?

- namenode는 보유한 meta data를 찾아서, 신호가 사라진 datanode 안 data들을 찾는다.
    - block들을 찾는 것이다.
- meta data에서 해당 노드를 지운다.
- 알고있는 rack 목록(rack awareness)에서 해당 노드를 지운다.
- block들을 가지고 있는 다른 node 중 하나에게 replica를 어떤 새로운 노드에게 복사하라고 명령한다.(re-replicating)
