# unsigned integer과 size_t

## iterator로서의 unsigned v.s. size_t

unsigned integer에 대해 c standard에서, ovf발생시 modular연산을 해야한다고 명시돼있다. 이를 이용해서 신뢰성 있는 프로그래밍을 할 수 있다고 한다. 여기서 한 발 더 나아가, size_t를 이용하면 더욱 신뢰성 있는 프로그래밍이 가능해진다. 

iterator로 사용할 i에 대해, 만일 unsigned int i였으면, malloc()으로 할당받을 공간을 넘어서거나, 모자란 경우가 생길 수 있다. 반대로 size_t를 이용하면, 충분한 공간을 가리킬 수 있게 된다. size_t는 word size (x86_64에서 64bits)의 unsigned integer이기 때문이다.
> 물론 종류가 이것저것 있다.
> - INT32_T, INT64_T, UNIT_64 등등....

더욱 구체적인 예를 들면, 한 객체가 메모리 상에서 A 만큼 크기를 할당받았다고 하자. 이때, iterator인 i는 unsigned int 타입으로 선언됐고, 시스템 환경에 따라 4B 크기를 가진다. 4B가 최대로 가리킬 수 있는 공간은 4GB 정도이고, A>4GB이라고 가정하자. 그렇다면 메모리 상의 객체를 iterator인 i가 모두 가리킬 수 없게 된다.

비슷한 예로, for loop의 조건문으로 적당한 크기의 cnt에 대해 ...; i <\cnt ; ... 가 있다고 하자. 이때, cnt가 unsigned MAX 값이어도 충분히 i의 범위 안에 들어간다.

## unsigned와 계산시 형변환

unsigned는 재밌는 성질이 많았다. 그 중 하나가, unsgined와 다른 정수 타입의 계산시, unsgined로 계산된다는 것이다. 예를 들면 이렇다. 
```c
unsigned ux=4;

if (ux > -2)
    ...
```
ux와 signed인 -2의 비교식은 unsigned가 있기 때문에 -2또한 unsigned계산이 되어서
0x4 < 0xFFFFFFF1110이 된다. 비극이 아닐 수 없다.
