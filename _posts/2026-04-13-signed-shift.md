요약: c에서 bit shift의 고려사항

# signed 와 shift(bits shift)연산

비트 연산시 꽤 재밌는 일이 많은데, 대부분 메모리 공간 크기 문제와 관련있다.

나타내고 싶은 수는 크고, 타입마다 할당해주는 메모리 공간은 유한하다.

특히나 c에서는 overflow라는 치명적인 사고를 강력히 막지는 않기 때문에, 비트 연산시 발생할 수 있는 문제들을 더욱 꼼꼼히 검토해야할 것 같다.

이번에는 그 중에서도 power-of-2 divide 연산(비트 연산자 <<, >>)에 대해서 공부한 내용을 정리하려고 한다.
- \>> 는 2의 n 제곱 나눗셈이다. 반대로 <<는 2의 n 제곱 곱셈이다. (하지만 늘 그렇지 않다는 문제가 있다. 아래에 나온다.)
- 단순히 / 로 나누기보다는 훨씬 빠른 연산이라고 한다. 하드웨어를 고려한 훌륭한 최적화 방법이지만, 실제론 요즘 컴파일러들이 / 로 쓰면 알아서 최적화 후 >> 로 바꿔준다고 한다.

우선, 짚고 넘어가야 할 게, c에서 메모리에 비트들을 담는 다는 원리는 같지만, 읽을 때 해석을 signed로 하냐, unsigned로 하냐에 따라 연산의 결과가 달라진다.

## Unsigned power-of-2 divide with shift

우선 unsigned numerical(부호가 없는 숫자 타입들)의 경우, **소수점을 버린다.**

```
int x = 15213 // 0b0011101101101101

x = x >> 4 // 950.8125,  x는 950 (0b0000001110110110)이 나온다.
```
unsigned shift라 logical shift가 된다. (엄밀히 c 표준에는 unsigned/signed 따라 logical sh냐, arithmatic sh냐 하는 내용이 없다고 한다. 다만 의례 통상적으로 컴파일러가 지원해준다고 한다.)

몹시 간단하다. 뒤에 4bits가 잘렸다. 수학적 함수를 쓰면, $\lfloor x \rfloor$ 가 되겠다.

그렇다면 signed에서는 다를까? 

## Signed power-of-2 divide with shift

우선, 수학적 식은 같다. $\lfloor x \rfloor$ 이다. 다만, 
- 10진수가 아닌, 비트 표현 - 2진수 라는 점이 다르고, 
- 또 '부호 비트'가 존재한다.

```
float x = -12,340.0 // 0b1100111111001100

x = x >> 4 // x는 -771.25가 아닌, 더 작은 정수인 -772로 떨어진다. (0b1111110011111100)
```
보면 오른쪽 bits 4개가 짤렸다.

문제는 `항상 낮은 방향으로 반올림`을 한다는 것이다. 

이 문제를 조금 다른 식으로 해석해보자면, n 비트의 signed 수는 다음과 같이 10진수로 표현할 수 있다.

$$-x_{w-1} \dot 2^{w-1}+\sum_{i=0}^{w-2}x_{i} \dot 2^{i}$$

즉, 양수 + 마지막 부호 비트자리의 여햐에 따른 -/+ 2^(w-1)이다. 실제 값은 `-771.25`였으니, 비트 sh 연산으로 잘려나간 소숫점 자리는 `0.75`였을 것이다.
> **양수값만큼 잃어 값이 음의 방향으로 커졌다!**

## 응용 문제

아마(가물가물하다) 과제로 `btest`라는 bit 연산자 문제들을 풀었었는데, 이 문제 중 하나가 "n번 bit shift 후 0에 가까운 방향으로 반올림 하시오."였다.
> 반올림이 항상 낮은 방향으로만 일어난다면 일종의 '편향'을 일으킬 수 있다. 
> - 데이터 값들이 조금씩 모자라게 편향된다면, 뭔가 후에 해석할 때 문제가 생길 수도 있을 것 같다. 

지금과 같은 결과로는 이 문제를 풀지 못한다. 

### 결론: 풀이

지금 원하는 건 0의 방향 반올림이다. 음수일 때는 더해지고, 양수일 때는 빼줘야 한다.

그렇다면, 어떤 크기의 수를 더해서, 양수일 때 빼더라도 더하나, 더하지 않으나 값이 같게 하고,

음수일 때는 오히려 이전보다 값이 커지는 방향을 찾으면 된다. 

예를 들어, 처음 예시로 들었던 >> 4 연산이라고 하자.
```
0b1100111111001100 를 네 칸 >> 하면, (-12,340)

0b1111110011111100(1100) 이다. (-772)

하지만 >> 연산 전에 0b11을 해주었다면?

0b1111110011111101 로 (-771)이 나왔을 것이다.
```
다음 예시도 같이 보자.
```
마찬가지로

0b1100111111001100 을 8 칸 >> 하면, (-12,340)

0b1111111111001111(11001100) 이 나온다. (-49) 원래 의도대로라면 -48.203125라 -48이 나와야 한다. (0 방향 반올림)

역시 이번에도 마찬가지로, 애초에 처음부터 0b11111111을 더했으면, 
 
0b1111111111010000 으로 -48이 나온다.
```
뭔가 감이 잡힌다. 아까는 0b1을 더해주고, 이번에는 0b11111111을 더해줬다.

이는 각각 `2^(k)-1`이다. (k는 right shift 횟수)
- 0b1은 2^(4)-1 
- 0b11111111은 2^(8)-1

그 이유는, **나머지가 조금이라도 있다면, 몫을 1 올려주기 때문이다.**
